Snake Game on FPGA with VGA Output

GitHub Repo

My team and I implemented a classic Snake game on an FPGA, driving a monitor via VGA. The system handles real-time input, rendering, game logic, and scorekeeping in Verilog. The design emphasizes clean module boundaries (timing, input, game state, renderer) and stable 60 Hz video output.

Role

RTL design, timing, input handling, testing.

Timeline

Nov–Dec 2024

Goals & Requirements

  • Render stable VGA frames at 640×480 @ 60 Hz with proper HSYNC/VSYNC timing.
  • Deterministic game loop: input → update → render per frame.
  • Snake movement, food spawning, growth on eat, and collision detection.
  • Score tracking and difficulty/speed scaling.
  • Glitch-free button input (debounced) and reset behavior.

System Architecture

The design is split into small, testable modules:

  • Clock & Timing: pixel clock generator; VGA timing (HSYNC/VSYNC, hcount/vcount, blanking).
  • Input: button/joystick interface with synchronous debouncing and edge detection.
  • Game State (FSM): idle → play → game-over; position FIFO for snake segments; RNG/LCG for food.
  • Renderer: color mux based on snake/food/background layers; optional score overlay.
  • Speed Control: tick divider to advance snake at level-dependent intervals.

Key Features

  • VGA controller supporting 640×480 @ 60 Hz (800×525 total timing).
  • Parametric grid size and snake length cap.
  • Difficulty tiers tied to score; speed ramps up over time.
  • Clean reset to known state; reproducible game start.
  • Optional pause and border-wrap or wall-collision modes (toggle at build-time).

Tech Highlights

  • Single-clock, synchronous logic; no inferred latches.
  • Two-process FSM style (sequential + combinational) for clarity.
  • RNG via Linear Congruential Generator seeded at reset.
  • Memory-efficient segment storage (circular buffer / FIFO pattern).

Implementation Notes

  • Timing: Pixel clock derived from board clock (e.g., 25.175 MHz or nearest achievable). HSYNC/VSYNC and porch intervals meet VESA spec.
  • Coordinate Map: (hcount, vcount) → board cell via integer division to avoid multipliers.
  • Collision: next-head vs walls/segments; enter game-over state on hit; update score on eat.
  • Rendering: priority: snake > food > background; simple 3-color palette for crisp pixels.
  • Testing: sim benches for VGA timing and FSM transitions; on-board LEDs for quick debug.

Results & Learnings

  • Stable 60 Hz output verified on external monitor; no tearing/flicker within porch margins.
  • Deterministic movement cadence and smooth difficulty scaling.
  • Takeaway: small composable modules + strict timing discipline = faster bring-up.

What I’d Improve Next

  • Sprite ROM and font overlay for UI and score.
  • Double-buffered grid for more complex animations.
  • Audio beeps via PWM/DAC on food and collisions.
